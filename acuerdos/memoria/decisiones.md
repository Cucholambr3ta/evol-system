# Decisiones clave

> Atomo de MEMORY. Decisiones de arquitectura y producto persistentes.

- **Registro de disciplinas 9→31** (2026-06-05, v0.3.2): el sistema integra 31 metodologias *-Driven (9 base + 22 extendidas), heredadas del upgrade X-DD `ultimate-update.md`. Registro canonico en `docs/disciplinas/INDEX.md`. 0 colisiones de ID.
- **Activacion por profile, no global:** cada proyecto declara su subset en `evol.profile.yml` (bloque `methodologies:`); el orquestador inyecta solo esas capas en su fase, resuelve el DAG de dependencias (ej. chaos exige odd_obs+threat_driven), y aplica los criterios como sub-gate. No se inyecta lo no declarado.
- **0 agentes permanentes nuevos:** las disciplinas se ejecutan con los 16 agentes core + efimeros (evol-agent-factory). El core permanente NO crece con el upgrade.
- **3 formas de integracion por disciplina** (campo `executor` en la ficha): mapeo a workflow existente, skill nueva (6 gaps: ux-driven, event-sourcing, api-versioning, iac-driven, debt-budget, use-case-driven), o capa declarativa (regla/sub-gate sin workflow).
- **Erradicación de Memoria Persistente (2026-06-06):** Se retiró Memoria Persistente y todo el "Modo COMPLETO" legacy de X-DD. La memoria queda gobernada por la Standard Library de Python en "Modo BASE" puro. Posteriormente se implementó EDMS completo (v0.6.0) con ChromaDB + NetworkX.
- **Skill readme-master como gate de calidad documental (2026-06-06, v0.5.0):** La skill `/evol readme-master` es el estándar oficial para cualquier archivo README.md del proyecto. Opera recursivamente sobre todos los subdirectorios y es activada automáticamente por el hook `pre-push` de `evol-gitflow.sh`. Los READMEs deben seguir el patrón Top 100 (HTML grid, Mermaid obligatorio, badges, sin emojis).
- **memory/ como directorio de runtime local (2026-06-06, v0.5.0):** La carpeta `memory/` (con `memory/raw/`) es un directorio de runtime local, ignorado por git por diseño. Debe ser creada por `evol-init.sh` en el bootstrap con permisos `750`. Nunca se versiona.
- **release-close como gate de memoria (2026-06-06, v0.5.0):** Todo release debe invocar `evol-gitflow.sh release-close --version=X.Y.Z` ANTES del tag git. Este comando dispara la actualización de `acuerdos/memoria/` y la sincronización de `MEMORY.md`. Sin este paso, la memoria persistente queda desactualizada respecto al estado real del repositorio.
- **Erradicación de GitNexus (2026-06-06):** Se eliminó GitNexus del framework completo (docs, configs, workflows). 17 archivos modificados, -435 lineas. Referencias residuales en `evol-agent-lifecycle.py` y `evol-evolve.py` protegidas por env var `EVOL_GITNEXUS=1` (inactivas por defecto). Razon: licencia PolyForm NC incompatible con uso comercial + requiere MCP.
- **Arquitectura EDMS: ChromaDB + NetworkX global (2026-06-06):** Memoria nativa implementada con ChromaDB (vector semántico) + NetworkX (graph/relacional) en una DB global `~/.evol/memory/` (no per-project). Separación por metadata, no por DB. 15 mejoras investigadas (24 repos analizados). Plan atómico en `docs/plan-implementacion-edms.md`. **v0.6.0 completado** — 6 fases, 41 tests, installer cross-platform.
- **Discipline-aware schema como diferenciador único (2026-06-06):** El schema de metadata de EDMS incluye `disciplinas[]` como multi-value filter — ningún otro framework de memoria maneja 31 disciplinas *-Driven como filtro de memoria. Habilita queries como "qué decisiones afectan SEGURIDAD?" o "qué lecciones de DDD hay?".
- **RRF fusion + composite scoring (2026-06-06):** Retrieval híbrido usa Reciprocal Rank Fusion (k=60) para fuscionar 3 streams (BM25 + Vector + Graph) con composite scoring (salience + recency + importance). Decay adaptativo por tipo de memoria. Referencia: agentmemory (95.2% R@5).
- **Auditoría e Higienización de Documentación (2026-06-06):** Se eliminaron emojis residuales (como `👉` en `MANUAL_USUARIO.md`) logrando un 0% de densidad de emojis en todo el repositorio. Se sincronizaron los 66 documentos y 14 dominios mediante `evol-doc-sync.py sync-all docs/` y `sync-folder` recursivo para corregir todos los drifts detectados.
- **Paridad y Espejo de Documentación en CLI (2026-06-06):** Se copiaron y sincronizaron los 11 archivos de documentación `.md` de raíz y sus sidecars `.json` en `src/evol_cli/docs/` para garantizar la paridad del paquete CLI distribuible.
- **EDMS Completo v0.6.0 (2026-06-06):** 6 fases implementadas: (1) `evol_memory_store.py` ChromaDB + NetworkX + stdlib fallback con auto-detect venv, (2) `evol-memory.py` 26+ subcomandos, (3) 4 lifecycle hooks, (4) 4-tier consolidation (raw→compressed→memory→knowledge), (5) FlowScript queries + team memory namespaces, (6) `edms-compact` con LLM + extractive fallback (Jaccard dedup + key sentences). `pyproject.toml` con optional-deps: memory, graph, full. 105 items bootstrapped, 63 drawers, 41 tests passing.
- **EDMS Bootstrap mejorado para proyectos existentes (2026-06-06):** `edms-bootstrap` detecta e importa: (1) acuerdos/**/*.md, (2) archivos legacy, (3) historial git, (4) disciplinas detectadas por archivos del proyecto, (5) nodos de Sprint existentes. Grafo se crea automáticamente con Proyecto, Branches, Disciplinas. `evol-init.sh` ejecuta `edms-bootstrap` al final del bootstrap.
- **EDMS Grafo auto-actualizable (2026-06-06):** `edms-index` ejecuta `_auto_update_graph()` que crea nodos (Proyecto, Sprint, Decision, Leccion, Riesgo, Handoff) y relaciones (DEFINE, TIENE, Genera, AFECTA, APLICA_A, Cierra). `sprint-close` crea nodo Sprint y indexa archivos de sprint.
- **Wireframes EDMS en acuerdos/design/wireframes/ (2026-06-06):** 3 wireframes HTML creados para UI: Global View (grafo + metricas + proyectos), Project Board (Gantt + Kanban + Stories + Metrics), Navigation Flow (flujo usuario + stack tecnico). Branch `feature/edms-ui` con `docs/edms-ui-STACK.md` (React 19 + FastAPI). Pendiente: aprobación de wireframes antes de implementar.
- **PyPI 0.6.0 publicado (2026-06-06):** `evol-dd 0.6.0` publicado exitosamente en PyPI via `twine upload`. Wheel: 899.5 kB, sdist: 1.0 MB. Tag `v0.6.0` apunta al merge de `feature/edms-memory-system` en develop. Versión pip global instalada localmente queda en `0.1.7` — actualizar con `pipx upgrade evol-dd` o `pip install --upgrade evol-dd`.
- **LadybugDB + ChromaDB como data stack principal (2026-06-06, v0.6.1):** LadybugDB (`pip install ladybug`, v0.17.1) es la graph database principal. ChromaDB es el vector store. NetworkX queda como fallback ligero. Grafo persiste en `~/.evol/memory/ladybug.lbug`. Schema: `MemoryNode(name STRING PRIMARY KEY, type STRING, properties STRING)` + `MemoryRel(FROM MemoryNode TO MemoryNode, relation STRING)`. Properties se almacenan como base64-encoded JSON para evitar que LadybugDB los interprete como Cypher map literals.
- **chromadb optional, no required (2026-06-06, v0.6.0):** `pip install evol-dd` funciona sin chromadb. `pip install evol-dd[memory]` o `[full]` agrega vector search. Fallback stdlib BM25 para proyectos sin ChromaDB.
- **Auto-detect venv en evol_memory_store.py (2026-06-06, v0.6.0):** Si chromadb no se encuentra en python del sistema, el script se re-ejecuta automáticamente con `.venv/bin/python3` donde ChromaDB está instalado. Transparencia total para el usuario.
- **Tier field en vez de tipo prefix (2026-06-06, v0.6.0):** Items se taggean con `tier=raw/compressed/memory/knowledge/archived` en metadata, no con prefijo en el texto. Decay adaptativo por tipo: decision=90d, leccion=30d, artefacto=14d, convencion=180d, riesgo=7d, handoff=1d, resumen=60d.
- **Extractive compaction fallback (2026-06-06, v0.6.0):** Cuando LLM no disponible, `edms-compact --tier` usa Jaccard dedup (threshold 0.7) + key sentence extraction (first sentence + sentences with important keywords). Ahorro ~50-75% de tokens sin LLM.
- **Cross-platform installer curl|bash (2026-06-06, v0.6.0):** `install.sh` detecta OS/Python/pipx automáticamente. Soporta Linux/macOS/WSL. Flags: `--no-ide` (solo framework), `--dry-run` (preview). Diseñado para usuarios no-Python que no tienen Python pre-instalado — instala Python via brew/apt si necesario.
- **Migración a MCP nativo (2026-06-06, v0.6.0):** Se abandonó la política anti-MCP. `evol.config.yml` tiene `mcp.enabled: true` por defecto; `evol-mcp.sh` (módulo `mcp-manager`) gestiona servers (add/remove/list/status); `evol-shield` ya no bloquea `mcpServers`. Residuos contradictorios corregidos: invariante en `evol.md` (era "Integración MCP en los adapters IDE" bajo "nunca violar"), comentario + verificación "0 refs MCP" en `evol-adapt.sh`. GitNexus se eliminó por licencia NC, NO por MCP.
- **Drift de VERSION corregido (2026-06-06):** `VERSION` estaba en 0.5.0 y `src/evol_cli/VERSION` en 0.4.0 vs `pyproject.toml`/tag = 0.6.0. Ambos sincronizados a 0.6.0.
- **Absorción de 5 capacidades externas (2026-06-06):** (1) anthropics/frontend-design → skill `evol-frontend-design` con doble `aesthetic_profile` (bold / enterprise-minimal); EDMS fija enterprise-minimal en `evol.profile.yml`. (2) upstash/context7 → skill `evol-context7` + MCP server registrado via `evol-mcp.sh add context7`. (3) snarktank/ralph → pattern `autonomous_loop` en `evol-orchestrate.py` (gate-guarded, reusa PLAN.md/memoria/state/gitflow). (4) obra/superpowers → CSO + RED-GREEN-REFACTOR añadidos a `crear-skill`. (5) remotion-dev → skill `evol-remotion` como capa de motion del toolkit de diseño (NO opt-in, parte del pipeline junto a evol-frontend-design). Atribuciones 7-11 en NOTICE. Catálogo regenerado: 14→17 skills.
- **Toolkit de diseño Evol-DD (2026-06-06):** `evol-frontend-design` (estética/tokens, qué se ve) + `evol-remotion` (motion/video, cómo se anima) forman el toolkit de diseño completo. Ambas se inyectan en Fase 4 (Build) y son combinables. El `aesthetic_profile` del profile gobierna la intensidad estética de ambas.
